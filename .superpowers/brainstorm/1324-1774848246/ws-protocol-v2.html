Message Protocol

Client → Server
{ type: "audio_chunk", data: base64 }
{ type: "audio_end" }
{ type: "barge_in" }
{ type: "mic_off" } ← NEW
{ type: "mic_on" } ← NEW
{ type: "pong" } ← NEW (heartbeat reply)
{ type: "disconnect" }
Server → Client
{ type: "state", value: "listening"|"thinking"|"speaking" }
{ type: "transcript", text: string }
{ type: "tts_chunk", audio: base64, text: string }
{ type: "tts_end" }
{ type: "engagement_prompt", reason: "silence"|"mic_off" } ← NEW
{ type: "ping" } ← NEW (heartbeat)
{ type: "interview_complete", summary: object }

New Behaviors

😶 Silence Engagement
  • Server tracks last audio_chunk time
  • After 8s silence in LISTENING: sends engagement_prompt (reason: "silence")
  • Client plays TTS: "Take your time — I'm listening whenever you're ready."
  • After another 15s: "Are you still there? Let me know when you're ready to continue."
  • After 30s total: session flagged as abandoned
🎤 Mic Off Detection
  • Client sends mic_off when candidate mutes
  • Server responds with engagement_prompt (reason: "mic_off")
  • Client plays TTS: "It looks like your microphone is turned off. Please unmute to continue."
  • Avatar shows a mic-off indicator on screen
  • On mic_on: silence timer resets, resumes LISTENING normally
🔗 Connection Health
  • Server sends ping every 10s
  • Client must reply pong within 5s
  • No pong → server closes socket + marks session as abandoned
  • Client watches WebSocket onclose: shows "Connection lost — reconnecting…" overlay
  • Auto-reconnect with same token (up to 3 attempts)
  • On reconnect: session resumes from last transcript position
