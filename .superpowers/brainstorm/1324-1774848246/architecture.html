Architecture: Real-Time Video Interview

WebSocket-based, replacing the current HTTP turn-based flow

System Flow

BROWSER (Frontend)
VRM Avatar (Three.js)
↕ lip-sync via AudioAnalyser

MediaRecorder (video/webm)
↓ continuous background recording

AudioWorklet (VAD)
↓ speech detected / end of speech

WebSocket Client
↕ audio_chunk / barge_in / audio_end
↕ tts_chunk / transcript / complete
⟷
WS
FASTAPI (Backend)
WS endpoint /ws/{token}

AudioBuffer collector
↓ audio_end signal received

faster-whisper (STT)
↓ transcript text

Ollama LLM (streaming)
↓ follow-up question text chunks

Piper TTS (streaming)
↓ audio chunks → client in real-time
📼 On interview end
Client uploads WebM recording → Supabase Storage
📄 Transcript + Eval
Saved to DB on session close → dashboard shows summary

What Gets Merged / Replaced

❌ Removed
  • POST /sessions/turn (HTTP)
  • POST /sessions/turn/audio (HTTP)
  • Turn-based audio submit loop
  • Base64 full audio in response
  • /conversational-interview route (merged in)
  • ConversationManager (replaced by WS handler)
→
✅ Replaced With
  • WS /ws/{token} (single persistent connection)
  • Streaming audio chunks over socket
  • Barge-in via {type:"barge_in"} message
  • TTS streamed back as binary frames
  • One unified /video-interview/[token] page
  • VideoInterviewWSHandler (new service)
