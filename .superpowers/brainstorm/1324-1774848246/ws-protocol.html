WebSocket Protocol & VAD / Barge-in

Message types, state machine, and how barge-in interrupts the AI

Message Protocol

Client → Server
// Continuous audio while speaking
{ type: "audio_chunk",
  data: base64string }

// VAD detected end of speech
{ type: "audio_end" }

// Candidate starts speaking while AI talks
{ type: "barge_in" }

// Client closed/left
{ type: "disconnect" }
Server → Client
// State transitions
{ type: "state", value: "listening"|"thinking"|"speaking" }

// Candidate's words (live feedback)
{ type: "transcript", text: string }

// TTS audio chunk (streamed, plays immediately)
{ type: "tts_chunk", audio: base64, text: string }

// All TTS chunks sent
{ type: "tts_end" }

// Interview finished
{ type: "interview_complete", summary: object }

Improved VAD (Voice Activity Detection)

❌ Current (Problems)
  • RMS-only threshold (0.03) — too sensitive to background noise
  • 1.5s silence timer — too long, feels sluggish
  • 100ms polling interval — CPU waste
  • Barge-in via separate HTTP call — awkward
  • No visual feedback on speech detection
✅ New (Improved)
  • AudioWorklet processes 20ms frames (not 100ms polls)
  • RMS + zero-crossing rate for better voice vs noise
  • Dynamic threshold: calibrates to ambient noise on start
  • End-of-speech: 800ms silence (down from 1.5s)
  • Barge-in threshold lower than speech-start threshold
  • Visual mic waveform shows detected speech level
