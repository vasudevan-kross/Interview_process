Real-Time Interview: Architecture Approaches

How should audio flow between the candidate and the AI interviewer?

A

WebSocket Streaming ★ RECOMMENDED

Upgrade video interview to full-duplex WebSocket. Client streams audio chunks continuously. Server does STT → LLM → TTS all streamed back. Avatar starts speaking as first TTS chunk arrives — no waiting for full response.

Pros

  • Truly real-time, low latency
  • Natural barge-in (send stop signal over socket)
  • Streaming TTS = avatar starts speaking in ~500ms
  • Works with existing Piper + Whisper stack
  • Recording still works (MediaRecorder runs in background)

Cons

  • Most complex to implement
  • Need to merge conversational interview into video interview
B

Streaming HTTP (SSE)

Keep turn-based flow but use Server-Sent Events to stream TTS audio chunks back as they're generated. Client still submits audio via POST, but response arrives in real-time chunks.

Pros

  • Simpler than WebSocket
  • Less refactoring needed
  • Avatar starts speaking faster than current

Cons

  • Still turn-based (not truly real-time)
  • Barge-in requires separate HTTP call (awkward)
  • Not "Google Meet" feel
C

Optimized HTTP (Fast Turn-Based)

Keep current HTTP approach but aggressively optimize: better VAD so submission is instant when candidate stops, parallel TTS generation, base64 audio back faster. Add barge-in via a separate abort endpoint.

Pros

  • Minimal architecture change
  • Easiest to implement

Cons

  • Still feels like "submit form, wait for response"
  • No streaming TTS — user waits for full audio
  • Barge-in is a hack, not real
  • Not the experience you described

My recommendation: Option A. The existing conversational interview already has WebSocket infrastructure — we merge it into the video interview flow, replacing HTTP turns with real-time streaming. The result feels like a real interview, not a form submission.
